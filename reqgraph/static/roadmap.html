1 · Parse & analyse a requirement Stable

IREB-Rupp / EARS role tiling with a byte-exact round-trip guarantee, quality smells, type and EARS classification, compound splitting.

Constraints

  • English only. The rule extractor's modality/condition cues (shall, when, while…) are English patterns.
  • Role boundaries are heuristic — unusual grammar (deep nesting, passive chains) can mis-tile a span.
  • Quality smells are lexical checks, not semantic understanding — a precise-sounding but wrong requirement passes.

Workarounds

  • Switch the backend (rules → spaCy → BERT) per requirement and keep the best tiling.
  • Edit the text and re-parse instantly; the round-trip check proves nothing was silently lost.
  • Compound inputs are auto-split into per-requirement tabs.

Planned

  • Human-in-the-loop tile correction that feeds the BERT tagger's training set (step 3 of the RAVEN process, made persistent).
  • Per-span confidence scores surfaced in the tiling view.
  • Additional language templates.

2 · Find connections in a set Stable

Links requirements that share a similar SUBJECT or OBJECT phrase; rendered as the interactive force-directed entity graph.

Constraints

  • Lexical similarity misses synonyms ("car" vs "vehicle" won't link without the embedding backend).
  • Pairwise scoring is O(n²) — instant for hundreds of requirements, slow beyond a few thousand.
  • The threshold is a manual dial; there is no auto-calibration.

Workarounds

  • Install the ml extra and switch similarity to BERT embedding for synonym-aware linking.
  • Lower the threshold to 0.3–0.4 to surface looser candidates, then judge them in the match table.
  • Split very large sets by subsystem before linking.

Planned

  • Approximate-nearest-neighbour index so 10k+ sets stay interactive.
  • Project glossary / synonym dictionary that boosts domain terms.
  • Cluster view: group requirements by shared entity automatically.

3 · Import & analyze a set Stable

CSV / Excel / JSON / ReqIF import with metadata columns, quality table, entity graph, and CSV/JSON/GraphML/Turtle/Cypher exports.

Constraints

  • ReqIF subset: string attributes only — enumeration values and XHTML-formatted text are not decoded.
  • Excel needs the io extra (pandas + openpyxl).
  • Everything is processed in one request in memory — a very large document blocks the single request thread.
  • No persistence: refresh the page and the analysis is gone.

Workarounds

  • Export enum/XHTML columns as plain strings before import (most RM tools can).
  • Download every artifact (CSV, JSON, GraphML…) immediately — they are the persistent record.
  • Split documents > a few thousand requirements into batches.

Planned

  • Full ReqIF datatype support incl. XHTML and enumerations.
  • Async processing with a progress bar for large documents.
  • Project save/load (SQLite) so sessions survive a refresh.
  • Direct DOORS / Polarion / Jama connectors.

4 · Compare a SysML v2 model Beta

Role-bucketed semantic match between SysML v2 textual notation and the requirement set: coverage both ways + gap lists.

Constraints

  • Textual notation only — no KerML interchange or SysML v2 API model-server access.
  • The regex parser reads the common constructs (part/action/state/ port/attribute/requirement); advanced ones (variant modeling, calc defs, allocations) are skipped.
  • Matching is name-based — shallower than the v1 path's neighborhood context.

Workarounds

  • For deeper matching, export the model as Turtle/RDF and use the SysML v1 card — its context-aware scorer works on any KG.
  • Keep element names descriptive; the score follows the words.

Planned

  • SysML v2 API/REST client — compare straight from a model server.
  • Port the v1 neighborhood-context scoring to the v2 path.
  • Feed embedded requirement blocks into the RVTM.

5 · Compare SysML v1 — XMI / Turtle / Cameo .mdzip Beta

Context-aware comparison (BFS neighborhood + satisfy bonus), ontology diff, KG exports, direct .mdzip upload, and custom-profile stereotype import with an override map.

Constraints

  • .mdzip extraction is validated on synthetic archives (standard uml_model.model layout + XML sniffing) — real Cameo exports vary by version and may hit edge cases.
  • No size cap on archive members yet — a pathological zip could exhaust memory (run locally, trusted files only).
  • Custom stereotypes are routed to roles by name heuristics; profile generalisation chains (e.g. ECU ◁ Block) are not followed because the profile usually lives in a separate module.
  • Turtle input needs rdflib; context building slows on very large models at 3 hops.

Workarounds

  • Pin ambiguous stereotypes exactly with the “Map your company profile” box (GUI) or --stereotype-map (CLI).
  • If a specific .mdzip fails, export XMI from Cameo (File → Export → Eclipse UML2 XMI) and use the XMI tab.
  • Reduce context hops to 1 for big models.

Planned

  • Profile-definition ingestion: resolve custom-stereotype inheritance automatically when the profile module is supplied.
  • Validation pass against real Cameo exports across versions + extraction size caps (zip-bomb guard).
  • Incremental parsing for very large models.

Certification traceability (RVTM) Heuristic — engineer confirms

Per-requirement Verified / Candidate / Gap trace status, IADT verification-method suggestion, findings list, CSV / Markdown / GraphML export.

Constraints

  • Verified requires exact requirement IDs in the model. By design, fuzzy text can never produce Verified — no IDs means no Verified rows, ever.
  • The IADT method (Test/Analysis/Demonstration/Inspection) is a keyword heuristic — a suggestion, not a verdict.
  • RAVEN is not a qualified tool (no DO-330 / tool-qualification pedigree); its outputs are engineering evidence, not certification credit by themselves.
  • Exports carry no signature or audit trail.

Workarounds

  • Make sure model requirements carry the same IDs as the requirement set — that single discipline unlocks auditable Verified rows.
  • Treat Candidate rows as your review queue; confirm or reject them in your RM tool.
  • Feed the RVTM CSV into your existing signed QA/CM process.

Planned

  • Human-in-the-loop review: confirm/reject Candidates in the GUI and persist the decision (Candidate → Confirmed).
  • Baselining + delta RVTM between model versions.
  • Signed/checksummed RVTM exports for evidence packages.

Force-directed entity graph Stable

Dependency-free physics layout of subject/object phrases with live view controls (colour-by, role filters, similarity floor, linked-only, labels).

Constraints

  • O(n²) simulation and SVG rendering — smooth to a few hundred nodes, not thousands.
  • Labels can overlap in dense clusters.
  • No image export of the graph itself yet.

Workarounds

  • Use linked-only + a higher min similarity to thin dense graphs; drag nodes apart and pin them.
  • Toggle labels off for overview, hover for details.

Planned

  • Canvas/WebGL renderer for thousands of nodes.
  • Cluster collapsing (one node per entity group).
  • PNG/SVG export of the current view.

Extraction backends (rules / spaCy / BERT) Stable

Pluggable role extraction: zero-dependency rules (default), spaCy grammar, or a trainable BERT token tagger.

Constraints

  • spaCy and BERT are optional installs (nlp / ml extras) — greyed out until installed.
  • The BERT tagger ships untrained; you must train it on your own labelled requirements (reqgraph train).
  • Inference on CPU is slower for large batches.

Workarounds

  • The rules backend is always available and is the tested default.
  • Train once on a few hundred labelled requirements; the tagger persists to a directory you can reuse via --model.

Planned

  • A pre-trained aerospace/automotive tagger shipped as a download.
  • ONNX export for lighter, faster inference without torch.

Platform & the path to production Local-first by design

Today RAVEN is a local, offline, single-user tool: a stdlib HTTP server bound to 127.0.0.1 serving one self-contained page. That is a feature for confidentiality (requirements never leave your machine) — and a constraint for teams.

Constraints

  • No authentication, HTTPS, rate limiting, or multi-user state — it must not be exposed beyond localhost as-is.
  • Blocking request handling: one heavy comparison stalls the UI.
  • No database — nothing persists between sessions.
  • The frontend is one large HTML file — fine to ship, harder to extend.
  • No upload size limits or archive extraction caps.

Workarounds

  • Run per-user locally (launch_raven.bat / python -m reqgraph gui) — the current supported mode.
  • Use the CLI for batch/CI jobs (export, compare, compare-v1 --rvtm) — it is fully scriptable and needs no server.
  • Keep artifacts (CSV/GraphML/Turtle) in your VCS as the durable record.

Planned — production architecture

  • API service: FastAPI + uvicorn, pydantic-validated requests, OpenAPI docs; current pure functions (parse_request, compare_v1_request…) become the service layer unchanged.
  • Jobs: background workers + progress events for BERT and large comparisons.
  • Persistence: SQLite→Postgres for projects, sets, review decisions, RVTM baselines.
  • Security: OIDC login, per-project authz, upload/zip caps, CSP; audit log for certification workflows.
  • Delivery: Dockerfile + compose, GitHub Actions CI (pytest, lint, coverage), versioned PyPI releases.
