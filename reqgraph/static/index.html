How RAVEN works — 4 steps

RAVEN takes a requirement set from raw document to a validated, machine-readable specification. The tools further down the page carry out steps 1, 2 and 4; step 3 is your review.

Use case — one drone, three engineers

Scenario: a team is specifying an autonomous inspection drone. The same requirement set flows through RAVEN's four steps, but each engineer owns a different slice of the loop. The sample files in samples/drone/ (a 4-level ReqIF set, a SysML v2 model and its knowledge graph) are exactly this story. Click an action to jump to its tool.

“A clean, atomic, testable baseline.”

  • Opens each requirement's IREB roles and confirms the graph regenerates the exact text — Parse.

  • Merges near-duplicates that share a subject/object — Find connections.

  • Compares the SysML v2 model to the baseline for semantic match, model coverage and requirement coverage — Compare v2.
  • For legacy Cameo/MagicDraw XMI, runs the context-aware, neighborhood-scored compare — Compare v1.
Surfaces: requirements with no design element, and design elements with no driving requirement (gold-plating).

“Proof each requirement is satisfied & verifiable.”
Owns steps 3–4

Together the three roles close RAVEN's loop: upload → check → validate → generate. The requirements engineer cleans the input, the architect proves the design covers it, and the V&V engineer validates and signs off the final, machine-readable specification.

3Import & analyze a set

The full pipeline over a whole document: parse every requirement, score its quality (weak words, atomicity, EARS pattern, type) and find cross-requirement connections. Upload a file (CSV / Excel / JSON / ReqIF) or paste text — extra columns like rationale or applicability are detected and kept in every download.

Accepts .csv .xlsx .xls .json .reqif .xml. CSV/Excel need a header row with a text column (and optionally id).
What do these settings mean?
  • Similarity — how two subject/object phrases are compared. lexical uses word overlap + character similarity (instant, no dependencies). BERT embedding compares meaning (catches synonyms) but needs PyTorch + transformers installed.
  • Threshold — the similarity cut-off, 0 to 1. Two requirements are linked only when a shared subject/object scores at or above this value. Higher = stricter (fewer, more confident links); lower surfaces more, looser matches. 0.6 is a balanced default.

What do the quality columns mean?
  • type — requirement category inferred from keywords: functional, performance, interface, safety, or usability.
  • ears pattern — EARS clause shape from the condition: ubiquitous (always active), event-driven (WHEN), state-driven (WHILE), unwanted behaviour (IF/THEN), or optional feature (WHERE).
  • weak words — vague/unverifiable terms (fast, robust, user-friendly, optimal, approximately…) that make a requirement ambiguous. Empty is good.
  • non-atomic — the sentence packs several actions joined by and/or; it should usually be split so each can be tested independently.
  • subject / object / condition / process / constraint — the parsed IREB-Rupp role spans extracted from each requirement; these drive the connections above (requirements sharing a similar subject/object link).
Other smells (passive voice, missing modality, vague quantifier, compound) are computed too and appear in the per-requirement view of the top panel.

4Compare a SysML v2 model

Check how well a SysML v2 model and your requirements line up. Model elements (parts, actions, states, attributes) are matched to the IREB roles in each requirement, giving a semantic match %, model coverage (elements traced to requirements), requirement coverage (requirements reflected in the model), and a gap list for both sides.

What do these settings mean?
  • Similarity — lexical compares word overlap and character patterns (instant, no dependencies). BERT embedding compares semantic meaning and catches synonyms (e.g. "decelerate" ↔ "slow down") but needs PyTorch + transformers installed.
  • Threshold — minimum similarity score (0–1) for a model element to count as matched by a requirement. Higher = stricter; 0.6 is a balanced default. Try 0.4 if results look too sparse.
  • Roles — which semantic role buckets to compare. SUBJECT = structural entities (parts, part defs); PROCESS = actions/behaviors (action defs); OBJECT = attributes/data; CONDITION = states and transitions.
  • Semantic match — F1-like harmonic mean of model coverage and requirement coverage. Penalises imbalance: 90% model + 10% req → only 18% match.
  • Model coverage — fraction of SysML model elements that appear in at least one requirement.
  • Requirement coverage — fraction of requirement semantic elements that are reflected in the model.

5Compare a SysML v1 model

For older SysML v1 models exported as XMI (MagicDraw, Cameo, Papyrus) or Turtle/RDF. Unlike step 4, each element is scored by its graph neighborhood, not just its name — so a block surrounded by telling parts and links matches better than its bare label. Pick the model format below, paste your requirements, and compare.

Upload a Cameo / MagicDraw .mdzip — the UML model is extracted from the archive and compared.
Roles
Custom stereotypes? Map your company profile

If your model uses custom-profile stereotypes (e.g. a SafetyRequirement block or a custom verifies relation), map each one to an IREB role (SUBJECT / OBJECT / PROCESS / CONDITION / ACTOR / CONSTRAINT) or a trace verb (satisfy / derive / refine / trace / allocate). One mapping per line — custom stereotypes are auto-detected, this just pins ambiguous ones.

How context hops work

Each model element is scored not just by its name but by a neighborhood context string: BFS up to N hops expands the element's local graph, concatenating neighbor names and documentation. A satisfaction bonus (0.20 weight) is added when the model already contains a satisfy/refine link from the element to a requirement whose text matches the input requirement. Higher hops → richer context but slower scoring.
